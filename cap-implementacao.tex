\section{Implementação do gerador de cursos colaborativos}
\label{sec:implementacaoGerador}

O gerador de cursos colaborativos é um software educativo que efetua a geração de unidades de aprendizagem adaptadas às características individuais dos estudantes. A geração acontece a solicitude de um projetista instrucional quem define os objetivos educacionais e participantes da unidade. Neste trabalho, a unidade de aprendizagem obtida consiste em um pacote ZIP que contém: um conjunto de roteiros colaborativos, definidos na especificação IMS-LD; e todos os recursos instrucionais necessários.


Uma vez obtido o planejador dependente de domínio, usando o processo de compilação apresentado na Subseção \ref{subsec:JSHOP2ip} e a modelagem do domínio de planejamento apresentada nos Capítulos \ref{cap:modelagem} e \ref{cap:modelagemEstrategias}, o gerador de cursos consiste apenas de um pequeno número de classes principais apresentadas no diagrama UML da Figura \ref{fig:classesJava}.

A classe \textsf{LDElement} representa a unidade de aprendizagem obtida pelo sistema de planejamento como um elemento de roteiro colaborativo que apresenta: 

\begin{itemize}
\item um atributo \emph{type} que define se o elemento de roteiro é uma etiqueta de abertura ou fechamento, atribuição ou remoção de um papel, adição ou remoção de um grupo, inserção de texto ou recurso;
\item um atributo \emph{tag} que define o nome da etiqueta;
\item um atributo \emph{parameters} que define a lista de atributos no marcador XML;
\item um atributo \emph{children} que define a lista de elementos que são parte do elemento; e
\item um atributo \emph{metadata} que define informações de classificação, objetivos educacionais e participantes conforme foi descrito na Subseção 4.3.4. Além disso, o atributo \emph{metadada} contém outras informações a serem definidas pelo gerador de curso como tarefas públicas, termos de busca, etc.
\end{itemize}

As classes encarregadas de efetuar o processo de planejamento instrucional são abstraias na interface \textsf{Solver}, interface que apresenta o método \textit{solve} para dar solução ao problema de planejamento instrucional (definido na Seção \ref{sec:problemaPlanejamento}). Assim, em nosso gerador de cursos a classe \textsf{JShop2ipSolver} encapsula o algoritmo de planejamento JSHOP2ip (detalhado na Subseção \ref{subsec:JSHOP2ip}) no atributo \textit{planner} e o domínio de planejamento (definido na Seção \ref{sec:dominioPlanejamento}) no atributo \textit{domain}.

A interface \textsf{IPublishHandler} apresentam métodos que possibilitam publicar as unidades de aprendizagem obtidas em um AVA. Assim, como métodos que manipulam a atribuição e remoção de usuários nos papéis e grupos. A classe \textsf{CCPublishHandler} implementa a interface para definir a integração do gerador de cursos com o software CopperCore \citep{vogten2004coppercore}. 

Finalmente, os métodos da classe \textsf{CourseGenerator} são empregados pelos projetista instrucional e outras ferramentas de autoria para efetuar a geração do curso mediante requisições de tipo GET e POST que empregam os métodos \textit{getTaskDefinitions} e \textit{generate}.

A lista de tarefas públicas <tasks> é utilizada pelo projetista instrucional ou outra ferramenta para definir o estado inicial \textit{initState} e a tarefa pública inicial \textit{initTask} como a requisição:
\begin{alltt}
[courseGeneratorBaseURL]/generate?_format=xml
   &initState=\textit{initState}
   &initTask.identifier=\textit{identifier}
   &initTask.goals[0].comp.id=\textit{compId\(\sb{0}\)}&initTask.goals[0].level=\textit{level\(\sb{0}\)}
   ...
   &initTask.goals[\(n\sb{1}\)].comp.id=\textit{compId\(\sb{n\sb{1}}\)}&initTask.goals[0].level=\textit{level\(\sb{n\sb{1}}\)}
   &initTask.skills[0].id=\textit{skillId\(\sb{0}\)}
   ...
   &initTask.skills[\(n\sb{2}\)].id=\textit{skillId\(\sb{n\sb{2}}\)}
   &initTask.attitudes[0].id=\textit{attitudeId\(\sb{0}\)}
   ...
   &initTask.attitudes[\(n\sb{3}\)].id=\textit{attitudeId\(\sb{n\sb{3}}\)}
   &initTask.groups[0].learners[0].id=\textit{learnerId\(\sb{0}\)}
   ...
   &initTask.groups[0].learners[\(m\sb{0}\)].id=\textit{learnerId\(\sb{m\sb{0}}\)}
   ...
   &initTask.groups[\(n\sb{1}\)].learners[0].id=\textit{learnerId\(\sb{m}\)}
   ...
   &initTask.groups[\(n\sb{1}\)].learners[\(m\sb{n\sb{1}}\)].id=\textit{learnerId\(\sb{m\sb{n\sb{1}}}\)}
\end{alltt}
Onde: 
\begin{itemize}
\item \textit{identifier} é um identificador de tarefa de planejamento instrucional pública (detalhados na Subseção \ref{subsec:redeTarefaInicial});
\item \textit{compId$_{0}$} ... \textit{compId$n_{1}$} são identificadores de competência cognitiva e \textit{level$_{n_{1}}$} ... \textit{level$_{n_{1}}$} são níveis de competência que definem objetivos educacionais \textit{goal$_{0}$} ... \textit{goal$_{n_{1}}$} (detalhados na Subseção \ref{subsec:objetivosEducacionais}); 
\item \textit{skillId$_{0}$} ... \textit{skillId$_{n_{2}}$} são identificadores de habilidades independentes;
\item \textit{attitudeId$_{0}$} ... \textit{attitudeId$_{n_{3}}$} são identificadores de atitudes; e
\item \textit{learnerId$_{0}$} ... \textit{learnerId$_{m_{0}}$}, ..., \textit{learnerId$_{m}$} ... \textit{learnerId$_{m_{n_{1}}}$} são identificadores de estudantes que definem os grupos de participantes que irão atingir os objetivos \textit{goal$_{0}$} ... \textit{goal$_{n_{1}}$}.
\end{itemize}

Então, o processo de geração continua efetuando o método \textit{generate} da classe \textsf{CourseGenerator} que efetua a criação da unidade de aprendizagem \textsf{e} (da classe \textsf{LDElement}) mediante o método \textit{solve}. Logo, o método \textit{toUoL} no objeto \textsf{e} da classe \textsf{LDElement} é empregado para obter uma lista de arquivos \textit{files} das unidades da aprendizagem descritas na especificação IMS-LD usando o algoritmo de geração de roteiros colaborativos detalhado na Seção \ref{sec:componenteConversao}.

Os métodos \textit{publishUoL} e \textit{createRunForUoL} são utilizados durante o processo de geração para publicar e criar ambientes de execução das unidades de aprendizagem usando a lista de arquivos \textit{files}. Assim, o metadado \textit{course} devolvido ao final do processo contém informações das URIs das unidades de aprendizagem e suas instancias de execução. \subsection{Mediador de consultas}
\label{subsec:mediador}

O componente mediador mostrado na Figura \ref{fig:mediatorcomponent} é utilizadas pelo domínio de planejamento no atributo \textit{domain} da classe \textsf{JShop2ipSolver} para dar solução aos termos de chamada que efetuam consultas nos modelos de domínio a ser ensinado e de estudantes. Assim, o mediador implementa as interfaces \textsf{BuildElements}, \textsf{GetElements}, \textsf{GetType}, \textsf{GetPropertyValue} e \textsf{GetRelated}.

\begin{figure}[htbp]
\centering 
\includegraphics[width=0.75\textwidth]{figuras/mediatorcomponent}
\caption{Visão geral do mediador dos modelos de domínio a ser ensinado e de estudante}
\label{fig:mediatorcomponent}
\end{figure}

As interfaces \textsf{QueryRepository} e \textsf{Register} no mediador são Serviços Web baseados em RESTful \citep{fielding2000representational} que efetuam consultas nos repositórios externos. Os repositórios são registrados mediante a interface \textsf{Register}, na qual são registrados o nome e a localização URL base do repositório. A interface \textsf{QueryRepository} possibilita ao mediador efetuar consultas nos repositórios mediante requisições de tipo GET ou POST na URL base do repositório definido como
\begin{alltt}
[repositoryBaseURL]/search?function=\textit{function}
   &\textit{key\(\sb{1}\)}=\textit{value\(\sb{1}\)}
   ...
   &\textit{key\(\sb{n}\)}=\textit{value\(\sb{n}\)}.
\end{alltt}

Se o termo da chamada é \textsf{(call BuildElement ?query [?id [?n]])} ou \textsf{(call GetElement ?query)} então o valor de \textit{function} é BuildElement ou GetElement; e os valores \textit{key$_{i}$}=\textit{value$_{i}$} são:

\begin{itemize}
\item types[i]=\textit{t} para cada termo \textsf{(class \textit{t})} da lista \textsf{?query};
\item properties[i].name=\textit{n} e properties[i].value=\textit{v} para cada termo \textsf{(property \textit{n} \textit{v})} da lista \textsf{?query};
\item relations[i].name=\textit{n} e relations[i].value=\textit{v} para cada termo \textsf{(relation \textit{n} \textit{v})} da lista \textsf{?query};
\item id=\textit{id} para o termo constante \textsf{?id}; e
\item nro=\textit{n} para o termo constante \textsf{?n}.
\end{itemize}

\begin{example}
O termo de chamada
\begin{alltt}\sffamily
     (assign ?result (call GetElements ((class Auxiliary)
                                                (property hasLearningResourceType (exercise diagram))
                                                (property hasLearningObjective (c1 s1k2))))
\end{alltt} 
é representado no repositório com URL base \textsf{http://repository.com}  mediante a requisição:
\begin{alltt}
http://repository.com/search?function=GetElements
    &class[0]=Auxiliary
    &property[0].name=hasLearningResourceType
    &property[0].value[0]=(exercise\%20diagram)
    &property[1].name=hasLearningObjective
    &property[1].value[0]=(c1\%20s1k2)
\end{alltt}
\end{example}


Se o termo da chamada é \textsf{(call GetType ?element)}, \textsf{(call GetPropertyValue ?element ?n [?dest])} ou \textsf{(call GetRelated ?element ?n)} então o valor de \textit{function} é GetType, GetPropertyValue ou GetRelated; e os valores \textit{key$_{i}$}=\textit{value$_{i}$} são:

\begin{itemize}
\item element[i]=\textit{e} para cada termo \textsf{e} da lista \textsf{?element};
\item name=\textit{n} para o termo constante \textsf{?n}; e
\item dest=\textit{d} para cada termo \textsf{d} da lista \textsf{?dest}.
\end{itemize}

\begin{example}
A função \textsf{(call GetPropertyValue aux1 hasLearningObjective comp1)}  é representado no repositório com URL base \textsf{http://repository.com} mediante a requisição:

\begin{alltt}
http://repository.com/search?function=GetPropertyValue
    &elements[0]=aux1
    &name=hasLearningObjective&dest[0]=comp1
\end{alltt}
\end{example}

\subsection{Integração do gerador de cursos com CopperCore}

O Coppercore \citep{vogten2006coppercore} é uma aplicação de código aberto em Java desenvolvida pela Universidade Aberta da Holanda que possibilita executar unidades de aprendizagem descritas na especificação IMS-LD. Assim, CopperCore é nosso motor de execução dos roteiros colaborativos obtidos pelo gerador de curso.

A classe \textsf{CCPublishHandler} (Figura \ref{fig:classesJava}) implementa a classe \textsf{IPublishHandler} para definir a integração do gerador do curso com o sistema CopperCore. O método \emph{publishUoL} nesta classe é encarregado de publicar e as unidades de aprendizagem obtidas pelo gerador de curso, unidades definidas como pacotes ZIPs (\textit{files}) que contém arquivos de manifesto ``\textit{imsmanifest.xml}'' e todos os recursos necessários.
